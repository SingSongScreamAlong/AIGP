## 2. Competition Context (What We're Building For)

### 2.1 Rules (from VADR-TS-001 + public sources)

- **Format:** Time-trial. Fastest valid time wins. All gates must be passed in order.
- **Compliance:** Any human interaction during a submitted run = immediate DQ.
- **Max run duration:** 8 minutes (plenty of head-room for slow-but-clean entries).
- **Prize pool:** $500K split among top 10. Highest-scoring entrant gets an interview skip-pass to Anduril hiring managers.
- **Registration:** Free. No credentials required. Teams of up to 8; we're 1.

### 2.2 Technical Interface (VADR-TS-001, issue 00.01, 2026-03-09)

- **Protocol:** MAVLink v2 over UDP via MAVSDK-compatible client
- **Physics rate:** 120 Hz
- **Recommended command rate:** 50–120 Hz
- **Minimum HEARTBEAT rate:** 2 Hz
- **Telemetry messages (sim → client):** HEARTBEAT, ATTITUDE, HIGHRES_IMU, ODOMETRY, TIMESYNC
- **Control messages (client → sim):** SET_POSITION_TARGET_LOCAL_NED, SET_ATTITUDE_TARGET
- **Coordinates:** Local Cartesian NED only. **No GPS, no geographic coords, no absolute global position.**
- **Vision:** Forward-facing first-person camera. Vision stream parameters are in a *separate spec not yet released*.
- **Runtime:** Python 3.14.2 known good; other envs allowed. Compiled extensions (C/Cython) permitted.

### 2.3 Hardware (Physical Phase)

- Identical Neros drones for all teams. 8-inch quadcopter class, ~2 kg payload capacity, up to 160 km/h.
- NVIDIA Jetson Orin NX compute (~100 TOPS). Comfortably runs YOLOv8-nano at 100+ FPS.
- Single monocular ~12MP wide-angle FPV camera. IMU only. **No LiDAR, no stereo, no depth.**
- Teams cannot modify hardware. Software-only competition.

### 6.2 Read of the results

- **Clean:** belief and vision tied. Neither improvement nor regression. Expected.
- **A3 periodic dropout:** belief wins when it completes (19.5s vs 21.5s, -2.0s). Belief propagation is the right tool for synchronized dropouts. 2/3 is unchanged from vision.
- **Harsh:** both fail identically. Belief isn't enough to rescue 15% miss + 5° bearing noise + 80° FOV. Other fixes needed (better detector, wider FOV assumption, front-back ambiguity resolution).
- **Mild:** **regression.** VisionNav gets 3/3 at 19.3s; BeliefNav gets 0/3, spending 83% of time in SEARCH mode. This is the critical failure.

### 6.3 Probable root cause (unverified)

`GateBelief.propagate()` at `gate_belief.py:94` is supposed to keep the body-frame bearing correct as the drone moves and yaws. The math:

1. Body-frame belief → NED offset using `yaw_rad`
2. Subtract `vel * dt` from NED offset
3. NED offset → body frame using `yaw_rad` (the **same** variable)

This correctly accounts for translation but **not rotation**. If the drone yaws between ticks, the body-frame bearing should counter-rotate. The stored `_prev_yaw_rad` is written but never read. So when the drone turns hard to approach a gate (exactly the case in mild-noise ≥25° turns), the propagated belief stays at its pre-turn body-frame bearing and rapidly drifts out of the actual FOV. Detection loss grows → confidence decays → belief dies → SEARCH → no reacquisition → timeout.

This is consistent with the "mild" condition specifically failing: detection losses are frequent enough to trigger propagation repeatedly during turns, but not so frequent that the drone never recovers (as in harsh).

### 6.4 Fix sketch (for whenever this is next picked up)

```python
def propagate(self, vel, yaw_rad, dt):
    prev_yaw = self._prev_yaw_rad if self._prev_yaw_rad is not None else yaw_rad
    cos_p, sin_p = math.cos(prev_yaw), math.sin(prev_yaw)  # for body→NED
    cos_n, sin_n = math.cos(yaw_rad), math.sin(yaw_rad)    # for NED→body

    # body→NED using PREV yaw
    body_x = self.range_est * math.cos(self.bearing_h) * math.cos(self.bearing_v)
    body_y = self.range_est * math.sin(self.bearing_h) * math.cos(self.bearing_v)
    body_z = self.range_est * math.sin(self.bearing_v)
    dn = body_x * cos_p - body_y * sin_p
    de = body_x * sin_p + body_y * cos_p
    dd = body_z
    # translate
    dn -= vel[0] * dt; de -= vel[1] * dt; dd -= vel[2] * dt
    # NED→body using CURRENT yaw
    new_body_x = dn * cos_n + de * sin_n
    new_body_y = -dn * sin_n + de * cos_n
    new_body_z = dd
    # … rest unchanged …
    self._prev_yaw_rad = yaw_rad
```

Also worth adding: a sanity check that `new_body_x > 0` (gate is in front), otherwise decay confidence faster — a belief behind the drone is meaningless in a forward-facing-camera regime.

This is the check that the whole detector→tracker→nav path actually works end-to-end on pixels. *Effort: 2 days. High info value: will expose inference-time latency and any frame-rate problems.*

### P1 — First 2 weeks after DCL sim drops

5. **Retrain the gate detector on DCL visuals.** Capture frames directly from the DCL sim, generate labels (either projected from known 3D or bootstrap with a loose first detector), fine-tune YOLOv8-nano. *Effort: 3–5 days. Determines Q1 competitiveness.*

6. **Plug in YOLOv8-pose keypoints + PnP.** We have both components; wire them together. This is the upgrade from "gate roughly here" to "gate opening exactly at 3D pose P." *Effort: 2–3 days. Required for ESKF and high-speed attitude control.*

7. **First Q1 submission.** Slow but reliable. Classical CV on highlighted gates if that works faster than YOLO — see BATTLE_PLAN §3 Q1 plan. Get a baseline on the official leaderboard. *Effort: 2–5 days after sim is stable.*

### P2 — Post-Q1 hardening (June/July)

8. **Error-State Kalman Filter** fusing PnP poses with IMU at 500 Hz. Replaces gate-relative-only tracker with smooth, high-rate position/velocity estimates. *Effort: 1 week.*

9. **Minimum-time trajectory optimizer** through gate sequence. Replaces direct-to-gate waypoints with curved paths that respect drone dynamics. *Effort: 1–2 weeks.*

10. **RL controller (PPO in DCL sim).** Train directly in the competition sim for zero sim-to-sim gap. This is the competitive edge. *Effort: 3–6 weeks of training + iteration. Only pursue if P0–P1 are clean.*

### P3 — Deferred / maybe never

- SLAM or full mapping — explicitly not needed.
- Stereo/depth estimation — we only have one camera.

- Custom sim. Use LSY now, DCL when it drops.
